# Deep audit — dambel-logo

Produce an evidence-based audit report. Report only; fix nothing.

## Read-only contract — non-negotiable

- **Do not modify, create, or delete any repo file.** The single exception is the report file itself (path below).
- **Do not run `generate.py` in a way that leaves changes behind.** Regeneration is the core reproducibility check, so it *is* allowed — but you must verify the tree is byte-identical afterwards and, if anything changed, **restore it** (`git checkout -- output/ README.md`) and report the drift as a finding. Take checksums before and after rather than trusting `git status` alone.
- Use the repo's pinned environment (`.venv`) rather than system Python — system Python is missing Pillow and cairosvg, and a different Pillow version can change output bytes.

## What to sweep

1. **Reproducibility (the headline check).** Checksum every file in `output/`, run `generate.py`, checksum again. Output must be **byte-identical**. If it is not, that is a `[blocking]` finding: either the committed output is stale relative to the generator, or generation is not deterministic. Report exactly which files changed. Restore the tree afterwards.
2. **Generator vs. committed output.** Every variant declared in `VARIANTS` has corresponding files in `output/` at every size in `SIZES`, and nothing in `output/` is orphaned by a variant that no longer exists. The `README.md` variant section (regenerated by the script) matches what the script actually produces.
3. **Brand color and gradient truth.** The `bgGradient` definition in each base template's `<defs>` must match what `CLAUDE.md` claims (colors *and* angle — a corner-to-corner `x1/y1 0% → x2/y2 100%` linear gradient is CSS `135deg`). Every template that uses the `gradient` color token must actually define `bgGradient`, or the token resolves to a broken `url(#bgGradient)`.
4. **Downstream brand consistency.** This repo is the source of truth for brand color. Check the documented values against the consumers and report divergence: `dambel-product-bible/design.md`, the app's `themes/app_colors.dart`, and the website's `globals.css`. Known standing divergence worth re-checking: the app derives gradients from `primary` with alpha rather than using the brand gradient's second stop. `banner.html` is a standalone file that deliberately uses a 90° horizontal variant — verify `CLAUDE.md` still documents that carve-out rather than contradicting it.
5. **Naming conventions.** The `<shape>_<bg-color>` scheme, the `_white` suffix forcing a white dumbbell, `dumbbell_*` being mark-only, `*_tight` using the cropped template, and the `background.svg → full_*` mapping. Report any output filename that does not follow the documented scheme.
6. **Claims in docs.** Every factual assertion in `CLAUDE.md` and `README.md` — dependency versions against `requirements.txt`, the described pipeline against `generate.py`, any CI/Dependabot/security claim against what is actually in `.github/` (or its absence).
7. **Hygiene.** Unused fragments in `assets/`, unused entries in `FRAGMENT_NAMES`/`COLORS`, stale `.gitignore` entries, and files in `output/` no longer reachable from any variant.

## Report

Write to `.claude/audits/<YYYY-MM-DD>-logo-audit.md` (create the directory if needed; it is gitignored). Structure:

- **Summary** — what matters, honestly ranked. This is a small repo; if it is clean, say so plainly in a few lines rather than manufacturing findings.
- **Reproducibility result** — literal: checksums matched or did not, and which files differed.
- **Findings**, grouped `[blocking]` / `[risk]` / `[nit]`, each with `file:line` evidence and why it matters.
- **Coverage** — what you checked and what you could not verify (e.g. whether a downstream repo was available to compare against).

Cite `file:line` for every claim. An honest "unverified" beats a confident guess.
